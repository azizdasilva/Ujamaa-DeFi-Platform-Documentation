1. Project Understanding

🏗️ What is Ujamaa DeFi Platform?

Ujamaa is an institutional-grade DeFi platform for tokenizing real-world assets (RWAs) in Africa. It enables:

  • Liquidity Pools (Pool Industrie, Pool Agriculture, etc.)
  • Yield-bearing tokens (uLP - Ujamaa Pool Token)
  • Collateral NFTs (UJG - Ujamaa Guarantee)
  • Compliance enforcement via ERC-3643 (T-REX protocol)
📊 Current Smart Contracts (MVP)
  • ULPToken.sol - Yield-bearing ERC-3643 token
  • LiquidityPool.sol - Pool management & financing
  • GuaranteeToken.sol - Collateral NFT (ERC-3643NFT)
  • IndustrialGateway.sol - Industrial onboarding
  • MockEscrow.sol - Bank escrow simulation
  • MockFiatRamp.sol - Fiat on/off ramp
✅ Target Users for Ujamaa Monitor

Non-technical stakeholders who need to understand platform health without blockchain expertise:

  • 🏦 Investors - Want to see pool performance and yield accrual
  • 🏭 Industrials - Want to track their financing status
  • 👥 Platform Admins - Need operational oversight
  • 📈 Regulators - Require compliance and audit trails

Component Breakdown

📡 Blockchain Layer (Polygon Amoy)

Purpose: Source of truth for all contract state and events.

  • RPC Endpoint: https://rpc-amoy.polygon.technology/
  • Contracts: Deployed MVP contracts (ULPToken, LiquidityPool, etc.)
  • Read Method: eth_call for state, eth_getLogs for events
🔧 Backend Layer (Optional but Recommended)

Purpose: Cache, aggregate, and simplify blockchain data.

  • Framework: FastAPI (reuse existing backend)
  • Location: frontend/src/services/monitor/
  • Functions:
    • Cache contract state (reduce RPC calls)
    • Calculate KPIs (NAV change, yield, etc.)
    • Index events into searchable format
⚠️ Architecture Decision: Backend vs Frontend-Only

Recommendation: Start with frontend-only (direct RPC calls) for MVP, then add backend caching if performance becomes an issue.

  • ✅ Frontend-Only Pros: Faster to build, simpler deployment, no backend costs
  • ❌ Frontend-Only Cons: Rate limits on RPC, slower page loads, no data aggregation

Decision: Build frontend-only first. Add backend if you hit RPC rate limits or need complex aggregations.

Real-Time Display Strategy

⚡ Live Updates (WebSocket Alternative)

Polygon doesn't support WebSocket subscriptions on public RPC. Use polling instead:

  • Critical KPIs: Poll every 5 seconds (NAV, pool balance)
  • Standard KPIs: Poll every 30 seconds (holder count, financing count)
  • Historical KPIs: Poll every 5 minutes (default rate, utilization)

5. Pedagogical Design Principles

🎓 Rule #1: Never Show Raw Blockchain Data

Transform every blockchain concept into a real-world analogy:

  • Smart Contract → "Automated Rule Book"
  • Transaction → "Action Record"
  • Gas Fee → "Processing Fee" (hide the wei/gwei complexity)
  • Block Number → "Timestamp"
  • Wallet Address → "Account ID" (show first/last 4 chars only)
📖 Tooltip Strategy

Every technical term gets a hover tooltip with:

  1. Simple definition (1 sentence)
  2. Real-world analogy
  3. "Why it matters" (impact on user)

Example:
NAV per Share ℹ️
"Current value of each uLP token. Like a stock price, but updates daily based on pool earnings."

📈 Visual Hierarchy

Use color and size to guide attention:

  • 🟢 Green = Good (yield earned, repayments made)
  • 🟡 Amber = Warning (high concentration, low liquidity)
  • 🔴 Red = Critical (defaults, compliance issues)
  • 🔵 Blue = Neutral/Info (new investor, minted UJG)
🎯 Progressive Disclosure

Don't overwhelm users. Show:

  1. Level 1: Big numbers (TVL, yield, holders)
  2. Level 2: Click for details (top investors, financing breakdown)
  3. Level 3: Advanced view (transaction history, contract addresses)
