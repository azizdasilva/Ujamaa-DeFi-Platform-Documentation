✅ Phase 2: Approve or Reject

3

Make Your Decision

What happens: After reviewing the documents, you approve or reject the application:

✅ Approval Flow:

  1. Click "Approve": The system validates the document is acceptable
  2. Database Update:
    • Document.verification_status changes from PENDING to APPROVED
    • InvestorProfile.kyc_status changes to APPROVED
    • Review timestamp and your officer ID are recorded
  3. Compliance Activity Logged:
    • A ComplianceActivity record is created with type DOCUMENT_APPROVED
    • Includes: reviewer ID, target document ID, timestamp, notes
  4. On-Chain Identity Verification (if blockchain enabled):
    • Backend calls IdentityRegistry.verifyIdentity(investor_wallet_address)
    • This writes the investor's verified status to Polygon Amoy
    • The investor can now hold ERC-3643 tokens (uLP)
// What happens on-chain (IdentityRegistry.sol) function verifyIdentity(address investor) external { require(hasRole(COMPLIANCE_OFFICER_ROLE, msg.sender)); Identity storage identity = s_identities[investor]; require(identity.status == IdentityStatus.PENDING); identity.status = IdentityStatus.VERIFIED; identity.verifiedAt = block.timestamp; identity.verifier = msg.sender; s_verifiedCount++; emit IdentityVerified(investor); }
✅ Investor Approved! The investor's KYC status is now APPROVED in the database. If blockchain is enabled, their wallet is also verified on-chain via IdentityRegistry. They can now invest in liquidity pools.

❌ Rejection Flow:

  1. Click "Reject": You must provide a reason (e.g., "Document unclear", "Address mismatch")
  2. Database Update:
    • Document.verification_status changes to REJECTED
    • review_notes field stores your rejection reason
    • InvestorProfile.kyc_status changes to REJECTED
  3. Investor Notification: The investor sees the rejection reason and can resubmit corrected documents
⚠️ Rejection is Reversible: If the investor resubmits better documents, you can review again and approve. Rejection doesn't permanently ban the investor.

⛓️ Phase 3: On-Chain Identity Management

4

IdentityRegistry On-Chain

What happens: The IdentityRegistry smart contract maintains on-chain identity verification for all investors:

🔐 Identity Lifecycle on Blockchain:

  1. Registration: IdentityRegistry.registerIdentity(wallet, jurisdiction, investorType)
    • Called during investor onboarding
    • Sets status to PENDING
    • Stores jurisdiction and investor type on-chain
  2. Verification: IdentityRegistry.verifyIdentity(wallet)
    • Called after compliance officer approves KYC
    • Sets status to VERIFIED
    • Records timestamp and verifier address
  3. Verification Check: IdentityRegistry.isVerified(wallet)
    • Called before every invest/redeem transaction
    • Returns true if identity is verified, false otherwise
    • If false, the transaction is blocked
  4. Revocation: IdentityRegistry.revokeIdentity(wallet)
    • Called if investor is found to be non-compliant after approval
    • Sets status to REVOKED
    • Investor can no longer hold or transfer uLP tokens
// Identity status transitions NONE → PENDING → VERIFIED → REVOKED ↓ REJECTED // What happens when investor tries to invest require(IDENTITY_REGISTRY.isVerified(investor_wallet), "ULPTokenizer: Recipient not verified"); // If not verified → Transaction reverts → Investment blocked
🔑 Why On-Chain Identity Matters:
  • Enforcement: Even if the database says an investor is approved, the smart contract will block them if their on-chain identity isn't verified
  • Auditability: Anyone can verify on Polygonscan which wallets are verified and when
  • Regulatory Compliance: Regulators can audit the full list of verified investors

📜 Phase 6: Asset Certification Review (NEW)

8

Review & Approve Asset Certifications

What happens: When industrial operators submit assets for financing, you review and approve the asset certifications:

📝 Asset Certification Flow:

  1. Operator submits asset: Industrial operator provides asset details and documentation
    • Asset type (Invoice, Inventory, Production, Shipment, Contract)
    • Declared value in EUROD
    • Supporting documents (invoices, warehouse receipts, contracts)
    • Warehouse location and quantity
  2. GDIZ/SIPI verifies: Physical verification authority inspects and certifies the asset
    • Physical inspection of warehouse/facility
    • Quality and quantity verification
    • Ownership confirmation
    • Upload verification to IPFS
  3. Compliance Officer reviews: You review the certification package
    • ✅ Verify GDIZ/SIPI certification is valid
    • ✅ Check asset valuation methodology
    • ✅ Confirm ownership documentation
    • ✅ Validate asset is not already encumbered
  4. On-chain recording: IndustrialGateway.certifyAsset() is called
    • Creates immutable Certificate record on Polygon Amoy
    • Links asset to industrial operator wallet
    • Stores IPFS hash for document integrity
// IndustrialGateway.sol - Asset Certification function certifyAsset( address industrial, string memory assetType, uint256 value, uint256 quantity, string memory unit, string memory warehouseLocation, bytes32 stockHash, string memory description, uint256 validityDays ) external onlyRole(CERTIFIER_ROLE) returns (uint256 certificateId) { certificateId = nextCertificateId++; certificates[certificateId] = Certificate({ industrial: industrial, assetType: assetType, value: value, quantity: quantity, unit: unit, warehouseLocation: warehouseLocation, stockHash: stockHash, description: description, certifiedAt: block.timestamp, expiresAt: block.timestamp + (validityDays * 1 days), isVerified: true, certifier: msg.sender }); emit AssetCertified(certificateId, industrial, assetType, value); return certificateId; }
✅ Asset Certified! The asset is now officially certified on-chain and can be used as collateral for financing. The next step is uGT minting when the financing is created.

🔒 ERC-3643 Compliance Flow

How Compliance Enforces Token Transfers

The ULPTokenizer contract has built-in compliance checks that prevent unauthorized transfers:

// ULPTokenizer.sol - Transfer override function transfer(address to, uint256 amount) public override returns (bool) { // ERC-3643: Check compliance before allowing transfer require( COMPLIANCE_MODULE.canTransfer(msg.sender, to, amount), "ULPTokenizer: Transfer blocked - " + COMPLIANCE_MODULE.getComplianceReason(msg.sender, to, amount) ); return super.transfer(to, amount); } // ComplianceModule.sol - Transfer validation function canTransfer(address from, address to, uint256 amount) public view returns (bool) { // 1. Check sender identity if (!IDENTITY_REGISTRY.isVerified(from)) return false; // 2. Check recipient identity if (!IDENTITY_REGISTRY.isVerified(to)) return false; // 3. Check jurisdiction restrictions if (BLOCKED_JURISDICTIONS[IDENTITY_REGISTRY.getJurisdiction(to)]) return false; // 4. Check investment limits if (amount > getDailyTransferLimit(from)) return false; return true; // All checks passed }
✅ Result: Even if someone tries to transfer tokens to an unverified wallet, the smart contract will automatically reject it. This is the power of ERC-3643 — compliance is built into the token itself.
💡 Key Takeaways:
  1. Database + Blockchain = Double Protection: Compliance status is tracked in the database (for analytics and reporting) AND on-chain (for enforcement)
  2. 24-Hour SLA: Every document must be reviewed within 24 hours — the system tracks and flags overdue items automatically
  3. IdentityRegistry is the Gatekeeper: If an investor's identity isn't verified on-chain, they cannot hold or transfer uLP tokens — no exceptions
  4. Complete Audit Trail: Every compliance action is logged with timestamp, officer ID, and details — enabling full regulatory reporting
  5. Sanctions Enforcement: Blocked jurisdictions are checked automatically — investors from sanctioned countries cannot onboard
